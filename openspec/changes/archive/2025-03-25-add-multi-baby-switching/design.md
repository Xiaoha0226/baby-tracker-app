## Context

当前宝宝记录应用使用 NestJS + TypeORM 后端和 Vue3 + Pinia 前端架构。用户系统已存在，但记录数据直接关联用户，没有宝宝维度。需要引入宝宝层级，让一个家庭可以管理多个宝宝的记录。

## Goals / Non-Goals

**Goals:**
- 支持一个用户创建和管理多个宝宝档案
- 提供直观的宝宝切换界面
- 所有记录、统计数据按当前选中宝宝隔离
- 保持现有用户体验，切换宝宝后页面数据自动刷新

**Non-Goals:**
- 多用户共享宝宝（权限系统）
- 宝宝成长曲线对比
- 宝宝之间的数据迁移/复制

## Decisions

### 1. 数据模型设计
**决策**: 新增 Baby 实体，Record 添加 babyId 外键
**理由**: 
- 清晰的一对多关系，一个用户可以有多个宝宝，一个宝宝有多条记录
- 符合现有架构风格，使用 TypeORM 关联

**替代方案**: 在 Record.details 中存储宝宝名称（简单但不利于查询和统计）

### 2. 当前宝宝状态存储
**决策**: 使用 Pinia store 存储 currentBabyId，localStorage 持久化
**理由**:
- 刷新页面后保持选中状态
- 全局可访问，方便 API 调用时自动附加 babyId

**替代方案**: URL 参数（每次切换需要修改路由，较繁琐）

### 3. UI 切换器位置
**决策**: 首页顶部 Header 区域，以下拉选择器形式展示
**理由**:
- 高频操作需要易于访问
- 不占用过多屏幕空间
- 与"统计"、"退出"按钮同层级，符合用户习惯

### 4. API 设计
**决策**: 在请求头或查询参数中传递 babyId
**理由**:
- 向后兼容：可选参数，不传递时保持原有行为
- 简单实现：不需要修改认证流程

**替代方案**: 路径参数 /api/babies/:id/records（需要大量修改路由）

### 5. 默认宝宝处理
**决策**: 用户首次使用时自动创建一个默认宝宝（如"宝宝"）
**理由**:
- 减少新用户的使用门槛
- 老用户数据迁移时可以关联到默认宝宝

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 老用户数据需要迁移 | 编写数据库迁移脚本，将现有记录关联到默认创建的宝宝 |
| API 频繁传递 babyId 容易遗漏 | 封装 API 请求拦截器，自动从 store 获取 currentBabyId |
| 切换宝宝时数据闪烁 | 添加加载状态，使用过渡动画 |
| 删除宝宝时关联记录处理 | 使用软删除或级联删除策略，添加确认提示 |

## Migration Plan

1. **数据库迁移**:
   - 创建 babies 表
   - records 表添加 baby_id 字段
   - 为每个现有用户创建默认宝宝
   - 将所有现有记录关联到默认宝宝

2. **后端部署**:
   - 新增 Baby 模块
   - 修改 Records 模块支持 babyId 筛选
   - 保持向后兼容（babyId 可选）

3. **前端部署**:
   - 新增宝宝管理页面
   - 首页添加切换器组件
   - 修改 API 调用传递 babyId

4. **回滚策略**:
   - 保留数据库备份
   - baby_id 字段可为空，回滚时忽略该字段
