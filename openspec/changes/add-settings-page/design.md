## Context

当前宝宝记录应用使用 Vue 3 + Pinia + Vue Router 构建前端，NestJS + TypeORM 构建后端。用户认证系统已实现，但缺少用户自我管理功能。用户目前无法修改自己的用户名或密码，这限制了账户管理的灵活性。

## Goals / Non-Goals

**Goals:**
- 提供直观的设置页面供用户管理账户信息
- 支持修改用户名（nickname）
- 支持修改密码（需要当前密码验证）
- 保持与现有 UI 风格一致
- 确保密码修改的安全性

**Non-Goals:**
- 不实现头像上传功能（avatar 字段已存在但本次不涉及）
- 不实现账户删除功能
- 不实现多因素认证
- 不修改用户登录名（username），仅修改显示名称（nickname）

## Decisions

### 1. 用户名修改策略
- **决策**: 允许修改 nickname（显示名称），不允许修改 username（登录名）
- **理由**: username 是登录凭证，修改可能导致混淆。nickname 用于界面显示，修改更安全
- **替代方案**: 允许修改 username - 拒绝，因为会影响用户登录体验

### 2. 密码修改验证
- **决策**: 修改密码时需要提供当前密码进行验证
- **理由**: 防止他人在用户离开设备时恶意修改密码
- **替代方案**: 直接修改 - 拒绝，安全性不足

### 3. 页面布局
- **决策**: 设置页面使用卡片式布局，分为"个人信息"和"修改密码"两个区域
- **理由**: 信息分组清晰，用户操作路径明确
- **替代方案**: 单页长表单 - 拒绝，视觉上过于冗长

### 4. API 设计
- **决策**: 使用 `PATCH /users/me` 更新资料，`POST /users/change-password` 修改密码
- **理由**: 符合 RESTful 设计，PATCH 用于部分更新，POST 用于特定操作
- **替代方案**: 使用 PUT - 拒绝，PATCH 更适合部分字段更新

### 5. 错误处理
- **决策**: 前端显示具体的字段级错误提示，后端返回统一的错误格式
- **理由**: 提升用户体验，帮助用户快速定位问题

## Risks / Trade-offs

- **[风险] 当前密码验证失败次数过多** → [缓解] 考虑后续添加速率限制，本次先实现基础功能
- **[风险] 密码修改后用户被登出** → [缓解] 修改密码后保持登录状态，但建议用户重新登录以验证
- **[权衡] 不修改 username** → 用户可能希望修改登录名，但为简化流程本次仅支持修改 nickname

## Migration Plan

无需数据库迁移，使用现有 users 表结构。

部署步骤：
1. 部署后端 API 更新
2. 部署前端页面更新
3. 验证设置页面可正常访问和操作
